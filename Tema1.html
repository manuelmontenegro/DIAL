1.1Especificación de algoritmos

Especificaciones informales

Escribir una función buscarElem que, dado un vector V de enteros, determine el índice, dentro del vector, en el que se encuentra un número entero x pasado como parámetro.

Por ejemplo, si V = [1, 6, -3, 5] y x = 6, la función devolvería 1.

 

Problemas

  • Si x aparece varias veces en V, ¿qué debe devolver?

    • ¿La primera aparición de x? ¿La última? ¿Cualquiera de ellas?

  • Si x no aparece en V, ¿qué debe devolver?

Especificaciones formales

  • Una especificación formal define el comportamiento de una función de manera precisa y sin dar detalles sobre su implementación.

  • Ventajas:

    👍 Elimina la ambigüedad del lenguaje natural.

    👍 Permite aplicar técnicas de razonamiento formal a los programas.

  • Utilizaremos especificaciones pre/post, basadas en el uso de precondiciones y postcondiciones.

Cabecera de las funciones

method nombre(x1: tipo1, …, xn: tipon) returns (y1: tipo1, …, ym: tipom)

  • Los parámetros pueden ser de tipo int, bool, real, o array<T> para un tipo T.
  • Si un parámetro o resultado es de tipo array, la descripción incluye la longitud de ese array.
    • Por ejemplo: V[0..n).
  • Si un parámetro es de entrada/salida, introducimos la palabra inout antes del nombre del parámetro.
  • En algunos lenguajes hay que hacer V.length, en otros hay que pasar explícitamente la n.

Ejemplo

method buscarElem(V[0..n): array<int>, x: int) returns (i: int)

Especificación pre/post de un método

method nombre(x1: tipo1, …, xn: tipon) returns (y1: tipo1, …, ym: tipom)
  requires P Precondición
  ensures Q Postcondición

  • P y Q son fórmulas lógicas.

  • P solamente puede mencionar los parámetros del método.

  • Q puede mencionar los parámetros y los resultados del método.

Precondición

method nombre(x1: tipo1, …, xn: tipon) returns (y1: tipo1, …, ym: tipom)
  requires P Precondición
  ensures Q

  • Indica los requisitos que han de cumplir los argumentos del método para poder llamarlo.

    • Si no hay requisitos, la precondición es la fórmula true.

  • Si los argumentos satisfacen la precondición, se garantiza que el método terminará devolviendo un resultado correcto.

  • Si los argumentos no satisfacen la precondición:

    • El método puede fallar durante su ejecución.
    • El método puede no terminar.
    • El método puede devolver un resultado incorrecto.

Precondición: ejemplos

method buscarElem(V[0..n): array<int>, x: int) returns (i: int)
  requires n >= 0

Indica que la longitud del vector V no es negativa.

method buscarElem(V[0..n): array<int>, x: int) returns (i: int)
  requires n >= 0 \({\wedge}\) (\(\exists\)i : 0 <= i < n : V[i] == x)

Indica que la longitud de V no es negativa, y que el elemento x se encuentra en el vector V al menos una vez.

Con esto ya, respondemos a alguna de las preguntas sobre qué pasa si el elemento no está en el vector.

Postcondición

method nombre(x1: tipo1, …, xn: tipon) returns (y1: tipo1, …, ym: tipom)
  requires P
  ensures Q Postcondición

  • Indica las propiedades que satisface la salida, con respecto a la entrada.
  • Si los argumentos de una llamada satisfacen la precondición P, el método garantiza que el resultado devuelto satisface la postcondición Q.

Postcondición: ejemplos

method buscarElem(V[0..n): array<int>, x: int) returns (i: int)
  requires n >= 0 \({\wedge}\) (\(\exists\)i : 0 <= i < n : V[i] == x)
  ensures 0 <= i < n \({\wedge}\) V[i] == x

Garantiza que el índice devuelto está dentro de los límites del vector, y que contiene el elemento buscado.
Si x aparece varias veces en V, no especifica cuál de las apariciones se devuelve.

 

method buscarElem(V[0..n): array<int>, x: int) returns (i: int)
  requires n >= 0 \({\wedge}\) (\(\exists\)i : 0 <= i < n : V[i] == x)
  ensures 0 <= i < n \({\wedge}\) V[i] == x \({\wedge}\) (\(\forall\)j : 0 <= j < i : V[j] != x)

Si x aparece varias veces en V, se devuelve el índice de la primera aparición.

Especificación alternativa de buscarElem

method buscarElem(V[0..n): array<int>, x: int) returns (i: int)
  requires n >= 0
  ensures -1 <= i < n
          \({\wedge}\) (i >= 0  ==> V[i] == x \({\wedge}\) (\(\forall\)j : 0 <= j < i : V[j] != x))
          \({\wedge}\) (i == -1 ==> (\(\forall\)j : 0 <= j < n : V[j] != x))

  • El método ya no exige que x pertenezca al vector.

  • Si devuelve un número positivo, es el índice de la primera aparición de x en V.

  • Si devuelve -1, entonces x no se encuentra en el vector.

Ejemplos: algoritmos numéricos

División de números naturales

La función divide, dados dos números naturales a y b, devuelve el cociente y el resto que resultan de dividir a entre b.

Primer intento

method divide(a: int, b: int) returns (q: int, r: int)
  requires a >= 0 \({\wedge}\) b > 0
  ensures a == b * q + r

Problema: un algoritmo que devuelva 0 como cociente y a como resto satisface la especificación.

La especificación no refleja la intención del programador/a.

Segundo intento

method divide(a: int, b: int) returns (q: int, r: int)
  requires a >= 0 \({\wedge}\) b > 0
  ensures a == b * q + r \({\wedge}\) 0 <= r < b

Dados a y b como en la precondición, solamente existe un par (q, r) que satisface la postcondición.

Raíz cuadrada real

method raiz(a: real) returns (r: real)
  requires a >= 0
  ensures r2 == a

method raiz(a: real) returns (r: real)
  requires a >= 0
  ensures r >= 0 \({\wedge}\) r2 == a

method raiz(a: real) returns (r: real)
  requires a >= 0
  ensures r >= 0 \({\wedge}\) |r2 - a| <= 0.0001

Raíz cuadrada entera

method raizEntera(a: int) returns (r: int)
  requires a >= 0
  ensures r >= 0 \({\wedge}\) r2 <= a < (r + 1)2

  • Dejar como ejercicio el máximo

Ejemplos: parámetros de entrada/salida

Intercambiar dos parámetros

La función intercambiar, dados dos parámetros de entrada/salida, intercambia sus valores.

Primer intento

method intercambiar(inout x: int, inout y: int)
  requires true
  ensures x == y \({\wedge}\) y == x

  • Cualquier algoritmo que asigne a x e y un mismo valor satisface la especificación.

  • ¿Cómo hacemos referencia a los valores que tienen x e y antes de ejecutar intercambiar?

x == 0 y == 0

Segundo intento

method intercambiar(inout x: int, inout y: int)
  requires x == xold \({\wedge}\) y == yold
  ensures x == yold \({\wedge}\) y == xold

  • Introducimos dos variables nuevas: xold,   yold.

    • Estas variables no son parámetros ni variables de programa; solo sirven para especificar.

  • En la precondición, decimos que xold e yold son los valores de x e y antes de ejecutar intercambiar.

Modificar los elementos de un vector

La función suma, dado un array de números enteros y un valor entero x, modifica el array sumando x a todos sus elementos.

method suma(inout V[0..n): array<int>, x: int)
  requires n >= 0 \({\wedge}\) V == Vold
  ensures (\(\forall\)i : 0 <= i < n : V[i] == Vold[i] + x)

1.2Implementación de algoritmos

Sentencias

S ::= skip;                          // No hacer nada
    | x := e;                        // Asignación
    | x1, …, xn := e1, …, en;        // Asignación simultánea
    | x1, …, xm := f(e1, …, en);     // Llamada a función
    | S1  S2                         // Composición secuencial
    | if b S1 else S2                // Condicional
    | while b S                      // Bucle

 

  • Denotamos variables por x, x1, y, z, etc.

    • Suponemos que cada variable ha sido declarada de antemano.

  • Denotamos las expresiones aritméticas por e, e1, etc.

  • Denotamos las expresiones booleanas por b, b1, etc.

Sobre la asignación simultánea

x1, …, xn := e1, …, en;

 

Evalúa los valores de las expresiones e1, …, en y después actualiza los valores de las variables x1, …, xn correspondientes.

Por ejemplo:

x, y := y, x;    Intercambia los valores de x e y

Tipos

  • Suponemos que el lenguaje tiene un sistema de tipos que asigna un tipo a cada variable y cada expresión.

  • Sintaxis de tipos:

    T ::= int
        | bool
        | real
        | array<T>

  • El sistema de tipos prohibe expresiones tales como 3 + true, v[2.5].

Funciones (métodos)

  • Utilizamos la misma sintaxis vista en la especificación de métodos.
  • Los métodos pueden incorporar una precondición y postcondición.

method nombre(x1: tipo1, …, xn: tipon) returns (y1: tipo1, …, ym: tipom)
  requires P
  ensures Q
{
  var z1: tipo’1, …, zk: tipo’k;  Variables locales
  S   Cuerpo del método
}

Declaración e inicialización

A veces inicializaremos una variable en su declaración:

var x: int := 5;

Equivale a:

var x: int; // en el inicio del método

…

x := 5;

Expresiones aritméticas

e ::= c     // literal (constante)
    | x     // variable
    | e1 + e2
    | e1 - e2
    | e1 * e2
    | e1 / e2
    | …
    | V[e]  // acceso a array

Dejamos abierto el conjunto de operadores aritméticos disponibles.

Expresiones booleanas

b ::= true        
    | false  
    | x
    | e1 == e2
    | e1 <= e2
    | …
    | b1 \({\wedge}\) b2
    | b1 \({\vee}\) b2
    | \(\neg\)b

  • Operadores relacionales habituales: ==, !=, <, <=, …

  • Los operadores \({\wedge}\) y \({\vee}\) utilizan evaluación de cortocircuito.

    • En la expresion b1 \({\wedge}\) b2, si b1 toma el valor false, no se evalúa b2.
    • En la expresion b1 \({\vee}\) b2, si b1 toma el valor true, no se evalúa b2.

    Esto nos permite expresiones como:

    0 <= i < n \({\wedge}\) V[i] > 0   // suponiendo V[0..n)

Ejemplo: división entera

method divide(a: int, b: int) returns (q: int, r: int)
  requires a >= 0 \({\wedge}\) b > 0
  ensures a == b * q + r \({\wedge}\) 0 <= r < b
{
    q := 0;
    r := a;
    while r >= b {
        q := q + 1; r := r - b;
    }
}

Ejemplo: intercambio de valores de parámetros

method intercambiar(inout x: int, inout y: int)
  requires x == xold \({\wedge}\) y == yold
  ensures x == yold \({\wedge}\) y == xold
{
  x, y := y, x;
}

Programas como transformadores de estados

Estados

  • Un estado \(\sigma\) es una función que asocia cada variable con un valor de su tipo correspondiente.

  • Por ejemplo: \([\texttt{x} \mapsto 3, \texttt{y} \mapsto 6]\).

Transformadores de estados en sentencias

  • Estado inicial: valores de las variables antes de ejecutar la sentencia.
  • Estado final: valores de las variables después de ejecutar la sentencia.

// Estado inicial: \(\sigma = [\texttt{x} \mapsto 3, \texttt{y} \mapsto 1]\)
x := x + 1; y := 2 * x;
// Estado final: \(\sigma' = [\texttt{x} \mapsto 4, \texttt{y} \mapsto 8]\)

  • Si \(S\) es la sentencia del ejemplo anterior, decimos que \(S\) transforma el estado \(\sigma\) en \(\sigma'\).

    Notación: \(\sigma \stackrel{S}{\rightsquigarrow} \sigma'\)

1.3Verificación de algoritmos sin bucles

¿Qué significa «verificar»?

  • Verificar formalmente un programa significa utilizar métodos matemáticos y lógicos para demostrar, de manera rigurosa, que la implementación de un programa cumple con su especificación.

  • No se basa en pruebas empíricas (tests), sino en razonamientos matemáticos.

Ventajas

  • Más garantías sobre la corrección de los programas, porque tiene en cuenta todos los casos posibles.
  • Permite detectar errores en fases tempranas de desarrollo.

Inconvenientes

  • Más costosa en términos de tiempo y recursos.
  • Requiere capacitación en matemáticas y lógica.

¿Qué necesitamos para verificar una función?

  1. La implementación de la función (es decir, una sentencia S).

  2. La especificación de la función (es decir, una precondición P y una postcondición Q).

Definición de función correcta (sin bucles ni recursión)

Dada la siguiente función con su especificación:

method f(x1: tipo1, …, xn: tipon) returns (y1: tipo1, …, ym: tipom)
  requires P
  ensures Q
{
  S    Suponemos que S no tiene bucles ni llamadas a f
}

  • La función es correcta si al ejecutarla en cualquier estado que satisfaga la precondición, se obtiene un estado que satisface la postcondición.

  • Es decir, para todo estado inicial \(\sigma\) y estado final \(\sigma'\) se cumple que:

    Si \(\sigma \vDash P\) y \(\sigma \stackrel{S}{\rightsquigarrow} \sigma'\), entonces \(\sigma' \vDash Q\)

Definición de sentencia correcta

Generalizamos estas nociones de corrección a sentencias. Cuando escribimos:

{ P } S { Q }

significa:

Si \(\sigma \vDash P\) y \(\sigma \stackrel{S}{\rightsquigarrow} \sigma'\), entonces \(\sigma' \vDash Q\)

Ejemplos:

{ x >= 1 } y := x * 2; { x >= 1 \({\wedge}\) y >= x }

 

{ 0 <= x <= 10 } x := x + 1; { 1 <= x <= 11 }

Técnicas de verificación de sentencias

Supongamos una sentencia S, una precondición P y una postcondición Q. ¿Cómo podemos demostrar lo siguiente?

{ P } S { Q }

  1. Lógica de Hoare

    Es un sistema de axiomas y reglas que permiten derivar ternas { P } S { Q }.

  2. Semántica de Transformación de Predicados (Dijkstra) Este tema

    Basada en la noción de precondición más débil.

  3. Otras técnicas: ejecución simbólica, etc.

Búsqueda de precondición

  • Dados una postcondición Q y un programa S, ¿qué precondición necesitamos para poder obtener lo siguiente?

    { ?? } S { Q }

  • En general, hay varios candidatos. Por ejemplo, con { ?? } x := y + 1; { x > 0 } tenemos, entre otras, las siguientes posibles precondiciones:

    { y >= 0 } x := y + 1; { x > 0 }

    { y > 0 } x := y + 1; { x > 0 }

    { y == 6 } x := y + 1; { x > 0 }

Fórmulas «fuertes» y «débiles»

  • Dadas dos fórmulas \(P\) y \(P'\). Si se cumple \(P \Rightarrow P'\), decimos que \(P\) es más fuerte que \(P'\) o, equivalentemente, \(P'\) es más débil que \(P\).

  • Las condiciones más fuertes son más restrictivas, en el sentido en que imponen más condiciones.

  • Por ejemplo, \(x > 0\) es más fuerte que \(x \geq 0\).

Fortalecimiento y debilitamiento

Proposición (fortalecimiento de la precondición):

Si { P } S { Q }, entonces se cumple { P’ } S { Q } para cualquier P' más fuerte que P.

Proposición (debilitamiento de la postcondición):

Si { P } S { Q }, entonces se cumple { P } S { Q’ } para cualquier Q' más débil que Q.

En general, nos interesan:

  • Las precondiciones débiles, porque imponen menos restricciones a la hora de ejecutar S.

  • Las postcondiciones fuertes, porque detallan mejor el estado final tras ejecutar S.

Búsqueda de la precondición más débil

  • Dados una postcondición Q y un programa S, ¿cuál es la precondición más débil P que permite obtener lo siguiente?

    { P } S { Q }

  • Definimos una función wp (weakest precondition) tal que wp(S, Q) devuelva la precondición P más débil que satisfaga { P } S { Q }.
  • Definimos la función wp por casos, según el tipo de sentencia S.

Definición de wp

Sentencia skip

wp(skip, Q) === Q

Si tras ejecutar skip queremos que se satisfaga Q, entonces necesitamos que también se satisfaga Q antes de la ejecución.

Definición de wp

Sentencia de asignación

wp(x := e, Q) === def(e) \({\wedge}\) Q[x/e]

Para que se cumpla Q tras la asignación x := e. Necesitamos que:

  • La expresión e esté bien definida.

    • def(e) es la fórmula que determina las condiciones necesarias para que e esté bien definida.

    • Por ejemplo: def(2 / x) === x != 0

  • Que la expresión e cumpla todo lo que Q afirma para la variable x.

    • La notación Q[x/e] refleja la sustitución de las apariciones libres de x en Q por e.

Ejemplo

{ ?? } x := y + 3; { x == 5 }

Calculamos wp(x := y + 3, x == 5):

wp(x := y + 3, x == 5)
    === def(y + 3) \({\wedge}\) (x == 5)[x/(y+3)]
    === true \({\wedge}\) y + 3 == 5
    === y == 2

Ejemplo

Suponiendo un vector V[0..n). ¿Qué precondicion necesitamos aquí?

{ ?? } x := x + V[i]; { x >= 0 }

wp(x := x + V[i], x >= 0)
    === def(x + V[i]) \({\wedge}\) (x >= 0)[x/(x + V[i])]
    === 0 <= i < n \({\wedge}\) x + V[i] >= 0

Inciso: dominio de definición de expresiones

def para expresiones aritméticas

def(c) === true              // donde c es una constante (número o booleano)
def(x) === true              // donde x es una variable

def(e1 + e2)  === def(e1) \({\wedge}\) def(e2)
def(e1 - e2)  === def(e1) \({\wedge}\) def(e2)
def(e1 * e2)  === def(e1) \({\wedge}\) def(e2)
def(e1 / e2)  === def(e1) \({\wedge}\) def(e2) \({\wedge}\) e2 != 0

def(V[e]) === def(e) \({\wedge}\) 0 <= e < n    // donde n es la longitud de V

Inciso: dominio de definición de expresiones

def para expresiones booleanas

def(true)  === true
def(false) === true
def(x) === true       // donde x es una variable booleana

def(e1 == e2) === def(e1) \({\wedge}\) def(e2)
def(e1 <= e2) === def(e1) \({\wedge}\) def(e2)
…
def(b1 \({\wedge}\) b2) === def(b1) \({\wedge}\) (b1 ==> def(b2))
def(b1 \({\vee}\) b2) === def(b1) \({\wedge}\) (\(\neg\)b1 ==> def(b2))
def(\(\neg\)b)      === def(b)

Definición de wp

Sentencia de asignación múltiple

wp(x1,…,xn := e1,…,en, Q) === def(e1) \({\wedge}\) … \({\wedge}\) def(en) \({\wedge}\) Q[x1/e1,…,xn/en]

Por ejemplo:

{ ?? } x, y := y, x; { x == 5 \({\wedge}\) y > 1 }

wp(x,y := y,x; , (x == 5 \({\wedge}\) y > 1))
=== def(y) \({\wedge}\) def(x) \({\wedge}\) (x == 5 \({\wedge}\) y > 1)[x/y,y/x]
=== true \({\wedge}\) true \({\wedge}\) y == 5 \({\wedge}\) x > 1
=== y == 5 \({\wedge}\) x > 1

Definición de wp

Asignación a posiciones de un vector

Suponemos que V tiene longitud n:

def(V[k] := e, Q) === def(e) \({\wedge}\) 0 <= k < n \({\wedge}\) Q[V/V’]

donde V' denota un vector de longitud n tal que:

  • V'[k] = e,

  • V'[i] = V[i] para todo i distinto de k.

Definición de wp

Llamadas a función

Supongamos que la función f tiene la siguiente especificación:

method f(x1: tipo1, …, xn: tipon) returns (y1: tipo1, …, ym: tipon)
requires P
ensures Q

Entonces:

wp(z1,…,zm := f(e1,…,en), R)
    === def(e1) \({\wedge}\) … \({\wedge}\) def(en) \({\wedge}\) P[x1/e1,…,xn/en] \({\wedge}\) (Q[y1/z1,…,yn/zn] ==> R)

Definición de wp

Composición de sentencias

{ ?? } S1  S2 { Q }

  • ¿Qué precondición necesitamos antes de ejecutar S2 para que se cumpla Q al final?

      wp(S2, Q)

  • Denotamos por R el resultado de wp(S2, Q) ¿Qué precondición necesitamos antes de ejecutar S1 para que se cumpla R justo antes de ejecutar S2?

      wp(S1, R)

  • Por tanto: wp(S1 S2, Q) === wp(S1, wp(S2, Q))

Ejemplo

{ ?? } z := x + 1; y := z - 1; x := y + 2; { x >= 0 }

wp((z := z + 1; y := z - 1; x := y * 2), x >= 0)
  === wp(z := z + 1, wp(y := z - 1, wp(x := y * 2, x >= 0)))
  === wp(z := z + 1, wp(y := z - 1, y * 2 >= 0))
  === wp(z := z + 1, (z - 1) * 2 >= 0)
  === (z + 1 - 1) * 2 >= 0
  === z >= 0

Definición de wp

Sentencias condicionales

{ ?? } if b { S1 } else { S2 } { Q }

  • Si se cumple la condición b, para asegurar Q al final de la ejecución, se ha de cumplir wp(S1, Q) antes de ejecutar S1.

  • Si no se cumple la condición b, para asegurar Q al final de la ejecución, se ha de cumplir wp(S2, Q) antes de ejecutar S2.

  • Por tanto:

    wp(if b S1 else S2, Q) === def(b) \({\wedge}\) (b ==> wp(S1, Q)) \({\wedge}\) (\(\neg\)b ==> wp(S2, Q))

Ejemplo

{ ?? } if y <= x { y := x; } else { skip; } { y >= x }

wp(if y <= x { y := x; } else { skip; }, y >= x)
  === def(y <= x) \({\wedge}\) (y <= x ==> wp(y := x, y >= x))
            \({\wedge}\) (y > x ==> wp(skip, y >= x))
  === true \({\wedge}\) (y <= x ==> x >= x) \({\wedge}\) (y > x ==> y >= x)
  === true \({\wedge}\) true \({\wedge}\) true
  === true

Verificación de sentencias

Volvamos al problema inicial. Dada una precondición P, una postcondición Q y una sentencia S sin bucles ni recursión.

Para demostrar que S es correcto con respecto a la especificación, es decir:

{ P } S { Q }

basta con demostrar:

P ==> wp(S, Q)

Ejemplo

method max(a: int, b: int) returns (m: int)
  requires true
  ensures m >= a \({\wedge}\) m >= b \({\wedge}\) (m == a \({\vee}\) m == b)
{
  if a > b { m := a } else { m := b }
}

Si P denota la precondición y Q denota la postcondición:

wp(if a > b { m := a } else { m := b }, Q)
 === def(a > b) \({\wedge}\) (a > b ==> wp(m := a, Q)) \({\wedge}\) (a <= b ==> wp(m := b, Q))
 === (a > b ==> Q[m/a]) \({\wedge}\) (a <= b ==> Q[m/b])
 === (a > b ==> a >= a \({\wedge}\) a >= b \({\wedge}\) (a == a \({\vee}\) a == b))
     \({\wedge}\) (a <= b ==> b >= a \({\wedge}\) b >= b \({\wedge}\) (b == a \({\vee}\) b == b))
 === true

Como P ==> true, la función es correcta.

Invariante de un bucle

Invariante

Supongamos un bucle while b { S }.

El invariante de un bucle es una fórmula lógica que se cumple:

  • Antes de la primera iteración.
    • Incluso antes de evaluar la condición del bucle (b) por primera vez.
  • Después de cada iteración.
    • Incluyendo la última iteración, antes de evaluar la condición del bucle por última vez.

El principal desafío de la verificación de programas con bucles es determinar los invariantes adecuados.

  • En el examen, en los ejercicios, etc.
  • No hay un procedimiento mecánico, requiere experiencia.

Corrección parcial de bucles while

¿Qué significa corrección «parcial»?

Supongamos un programa S (con bucles y/o recursión), una precondición P y una postcondición Q.

Decimos que S es parcialmente correcto con respecto a la especificación (P, Q),

{ P } S { Q }

cuando al ejecutar S sobre un estado \(\sigma\) que satisfaga P, la ejecución termina en un estado \(\sigma'\) que satisface Q, o bien no termina.

Formalmente:

Si \(\sigma \vDash \texttt{P}\) y \(\sigma \stackrel{\texttt{S}}{\rightsquigarrow} \sigma'\), entonces \(\sigma' \vDash \texttt{Q}\)

La condición \(\sigma \stackrel{\texttt{S}}{\rightsquigarrow} \sigma'\) supone implicitamente que la ejecución de S termina.

Corrección parcial de bucles while

Para determinar que un bucle while satisface una especificación:

{ P } while b invariant I { S } { Q }

Hay que demostrar cuatro cosas:

  1. El invariante se cumple antes de realizar la primera iteración: P ==> I

  2. La condición del bucle está bien definida en cada iteración: I ==> def(b)

  3. El invariante se preserva en cada iteración del bucle: { I \(\wedge\) b } S { I }

  4. Al salir del bucle, se cumple la postcondición: I \(\wedge\) ¬b ==> Q

{

  i := i + 1;

1.5Verificación práctica de programas con bucles

Bucles dentro de programas

{ P } while b invariant I { S } { Q }

Podemos tener sentencias antes y después del bucle

{ P } S1  while b invariant I { S }  S2 { Q }

¿Qué condiciones intermedias se cumplen antes y después del bucle?

{ P } S1 { ?? } while b invariant I { S } { ?? } S2 { Q }

Condiciones intermedias

{ P } S1 {  I  } while b invariant I { S } {  wp(S2, Q)  } S2 { Q }

Tenemos que demostrar:

  1. P ==> wp(S1, I)

  2. { I } while b invariant I { S } { wp(S2, Q) }

1.6Terminación de bucles

Corrección total

Supongamos un programa S (con bucles y/o recursión), una precondición P y una postcondición Q.

Decimos que S es totalmente correcto con respecto a la especificación (P, Q),

{ P } S { Q }

cuando al ejecutar S sobre un estado \(\sigma\) que satisfaga P, la ejecución siempre termina, y lo hace en un estado \(\sigma'\) que satisface Q.

Formalmente:

Si \(\sigma \vDash \texttt{P}\), entonces existe un \(\sigma'\) tal que \(\sigma \stackrel{\texttt{S}}{\rightsquigarrow} \sigma'\) y \(\sigma' \vDash \texttt{Q}\)

Corrección total

Corrección total \(=\) Corrección parcial \(+\) terminación

Funciones de cota
(ranking functions)

Función de cota en un bucle

Una función de cota es una expresión que depende de las variables del bucle y que cumple las siguientes condiciones:

  1. Mantiene un valor ≥ 0 antes de cada iteración del bucle.

  2. Decrece estrictamente cada vez que realizamos una iteración del bucle.

 

Encontrar una función de cota para un bucle dado implica demostrar que el bucle termina.

El valor de esta función en el estado inicial proporciona una cota superior del número de iteraciones que realiza el bucle.

Ejemplo

{ n >= 0 }
i := n; s := 0;
while i != 0 do
  invariant 0 <= i
{
  s := s + 1;
  i := i - 1;
}

Función de cota: i

  • Según el invariante, i siempre es mayor o igual que cero.
  • En cada iteración decrementamos i, por lo que i disminuye.

Ejemplo 2

{ n >= 0 }
i := 0; p := 1;
while i != n do
  invariant i <= n
{
  p := p * 2;
  i := i + 1;
}

Función de cota: n - i

  • Según el invariante tenemos i <= n que implica n - i >= 0
  • En cada iteración aumentamos i, por lo que n - i disminuye estrictamente.

Anotación decreases

  • Si queremos hacer explícita la función de cota de un bucle, la podemos indicar en los bucles:

i := 0; p := 1;
while i != n do
  invariant i <= n
  decreases n - i   Función de cota
{
  p := p * 2;
  i := i + 1;
}
